# 小红书私信助手产品规划

## 1. 现状判断

当前项目的真实可用能力不是 Web 端，而是 Android 真机链路。

现阶段需要明确这几个事实：

- 小红书网页版目前无法承载私信主流程，`web` 通道只能作为后续接口抽象保留，不能计入当前产品能力。
- 当前唯一可执行的主链路是 Android 通道，通过 `adb`、UI dump 和界面解析识别会话与消息。
- 当前 MVP 不是“自动回复机器人”，而是“私信发现 + AI 草稿生成 + 飞书通知 + 人工发送”的辅助系统。
- 近期研发重点不是扩功能，而是先把 Android 链路打通并稳定下来，然后再开发自动发送。

因此，这个产品当前的定位不应该是“网页私信助手”，而应该是：

**基于 Android 真机执行的小红书私信 AI 辅助系统**

## 2. 当前 MVP 定义

### MVP 目标

先把一条能真实使用的私信处理链路跑通，并在真实设备上稳定工作。

### 当前 MVP 范围

- Android 真机或模拟器连接
- 唤起小红书 App 并进入消息页
- 读取会话列表和最近消息上下文
- 识别未读或待处理会话
- 调用大模型生成回复草稿
- 将用户消息、建议回复、处理结果推送到飞书
- 由人工在手机上完成最终发送

### 当前 MVP 不包含

- Web 私信处理
- Android 自动发送
- 多账号调度
- 管理后台
- 客户资料管理
- CRM、知识库、工单系统
- 纯云端 SaaS 部署

## 3. 产品目标

这个项目的短期目标不是“做出功能最多的版本”，而是先解决两个问题：

1. 不漏掉私信。
2. 让运营人员更快给出一条合适的首轮回复。

从对外交付角度看，真正有价值的不是“AI 自动发了多少条”，而是：

- 私信有没有被及时发现
- 草稿是否足够可用
- 整条链路是否稳定
- 外部客户能否安装、理解、维护和持续使用

## 4. 目标用户

### 第一阶段目标用户

- 1 到 3 个小红书账号的小团队
- 有明确私信承接需求的个人创作者
- 愿意提供一台专用 Android 设备的外部客户
- 能接受“AI 草稿 + 人工发送”工作方式的运营人员

### 暂不建议覆盖的用户

- 希望开箱即用、完全不碰设备的纯小白用户
- 一开始就要求大规模矩阵号统一运营的团队
- 期望项目天然替代 CRM 或完整客服系统的客户

原因很简单：当前产品的执行层是 Android 真机，不是云 API。交付模式必须与这个事实一致。

## 5. 产品定位

### 定位语

小红书私信助手是一套面向创作者和轻运营团队的 Android 真机私信辅助系统，用于发现新私信、生成 AI 回复草稿，并通过飞书把处理结果实时通知给运营人员。

### 核心价值

- 降低漏回私信的概率
- 缩短首轮回复时间
- 降低重复回复的人工成本
- 在不贸然自动发送的前提下，让 AI 先参与价值最高的一段流程

### 当前不主打的价值

- 完全自动化
- 大规模账号管理
- 多人协作后台
- 精细化销售管理

## 6. 当前产品能力与缺口

### 已有能力

- Android 通道可连接设备、启动 App、抓取 UI 结构
- 可以解析消息列表与会话上下文
- 可以基于消息内容生成回复草稿
- 可以把结果通知到飞书
- 有本地状态和基础去重机制
- 有 Android UI 调试与解析相关测试

### 当前主要缺口

- Android 发送链路尚未实现
- UI 解析稳定性仍需持续验证
- 缺少设备异常、断连、页面偏移的恢复机制
- 缺少面对外部客户的安装和排障体验
- 缺少任务历史、审计记录和运行状态可视化

这意味着当前最大的风险不是“AI 不够聪明”，而是“设备链路不够稳”和“客户无法低成本使用”。

## 7. 产品原则

1. Android 真机链路优先于 Web 抽象。
2. 稳定性优先于自动化。
3. 草稿可用性优先于模型花哨度。
4. 单账号闭环优先于多账号扩展。
5. 对外交付时，设备管理能力必须被纳入产品设计，而不是留给客户自己解决。

## 8. 面向外部客户的交付判断

如果按未来可交付产品来规划，这个项目不能按“纯代码仓库 + README”思路继续往前推。

外部客户真正需要的不是源码，而是：

- 一套可安装的运行方式
- 一台可绑定的 Android 设备
- 一个清晰的通知与操作入口
- 出问题时能看懂的状态和错误信息

所以，当前阶段对外交付的合理方向应该是：

**本地 Agent / 专用主机 + Android 真机 + 飞书通知**

而不是：

**纯 Web SaaS**

### 推荐的首个交付形态

第一版面对外部客户，建议采用：

- Windows 本地 Agent 或专用小主机常驻服务
- 一台专用 Android 手机作为执行设备
- 飞书作为通知和观察入口
- 人工在手机端完成最终发送

这个形态虽然不够“轻”，但与当前项目现状一致，能真正落地。

## 9. 阶段路线图

### Phase 0：链路打通与稳定

目标：把 Android 真机读取链路稳定下来，形成可日常运行的草稿模式。

范围：

- 设备连接校验
- 小红书页面识别与会话提取
- 最近消息上下文解析
- AI 草稿生成
- 飞书通知
- 本地状态存储与去重
- 基础日志和错误分类

阶段标准：

- 在固定设备上连续运行 3 到 7 天
- 不频繁漏消息
- 不出现明显重复处理
- 常见故障可通过日志定位

### Phase 1：内部可运营版本

目标：让内部团队用真实账号稳定跑起来。

范围：

- 优化会话识别准确率
- 提升飞书通知的信息密度
- 增加设备断开、页面异常、App 不在目标页时的报错提示
- 形成标准安装与调试流程
- 明确“人工发送”操作规范

阶段标准：

- 一个真实账号可持续使用
- 运营人员能根据飞书通知完成处理
- 项目可以被内部非开发成员接手运行

### Phase 2：自动发送 Beta

目标：在 Android 端实现受控自动发送。

范围：

- 定位输入框与发送按钮
- 草稿写入与点击发送
- 发送结果确认
- 敏感关键词转人工
- 冷却时间和重复发送保护
- 失败重试和异常回退策略

阶段标准：

- 在低风险场景中稳定发送
- 误发送率可控
- 发送成功与失败都有清晰记录

### Phase 3：外部客户试点交付

目标：把现有系统打包成一个可交付给外部试点客户的产品。

范围：

- 客户安装包或部署手册
- 设备绑定流程
- 运行状态检查
- 基础任务历史与审计记录
- 升级与故障支持流程

阶段标准：

- 外部客户在你们指导下可以完成部署
- 一台设备绑定一个账号的模式可持续运行
- 支持成本可控

### Phase 4：正式产品化

目标：把系统从“可交付”推进到“可规模复制”。

范围：

- 设备管理面板
- 多客户运行状态总览
- 远程日志与告警
- 简化安装流程
- 桌面端或轻量控制台

阶段标准：

- 新客户部署效率提升
- 运行问题可以远程定位
- 不再强依赖开发者人工值守

## 10. 功能优先级

### 现在必须优先做

- Android 消息列表识别稳定性
- 会话上下文解析准确率
- 设备连接与异常恢复
- 飞书通知质量
- 日志、错误提示与排障手册

### 接下来应该做

- Android 自动发送
- 发送确认与防误发机制
- 任务历史和审计记录
- 安装包或半自动部署方案
- 设备状态监控

### 后续再做

- 多账号支持
- 管理后台
- 知识库增强
- 客户标签和线索管理
- Web 通道重新接入

## 11. 当前 RICE 优先级建议

| 功能 | Reach | Impact | Confidence | Effort | 优先级 |
| --- | ---: | ---: | ---: | ---: | --- |
| Android 链路稳定性 | 8 | 3 | 0.9 | 3 | 最高 |
| 设备异常恢复与日志 | 8 | 3 | 0.85 | 3 | 最高 |
| 飞书通知与人工处理闭环 | 7 | 2 | 0.9 | 2 | 高 |
| Android 自动发送 | 7 | 3 | 0.7 | 5 | 高 |
| 安装部署与外部交付文档 | 6 | 3 | 0.8 | 3 | 高 |
| 任务历史与审计记录 | 5 | 2 | 0.75 | 4 | 中 |
| 多账号支持 | 3 | 2 | 0.6 | 6 | 低 |
| Web 通道恢复接入 | 2 | 2 | 0.4 | 5 | 当前低 |

结论：

- 现在最该投入的是 Android 稳定性，不是 Web。
- 自动发送很重要，但必须建立在读取、识别、通知稳定之后。
- Web 端应保留抽象，但不应该进入当前排期核心。

## 12. 成功指标

### 当前阶段核心指标

- 新私信发现成功率
- 从收到消息到生成草稿的时间
- 草稿可采纳率
- 飞书通知送达率
- 设备在线时长
- 单日异常中断次数

### 自动发送阶段新增指标

- 自动发送成功率
- 自动发送误发率
- 自动发送回退到人工的比例
- 同一会话重复发送率

### 外部交付阶段新增指标

- 单客户部署耗时
- 客户自主恢复常见问题的比例
- 单客户周支持次数

## 13. 推荐产品架构方向

按当前状态和未来交付要求，建议架构方向为：

### 执行层

- Android 真机
- `adb` 通信
- UI dump 与界面操作

### 业务层

- 会话发现
- 上下文提取
- 风险策略
- AI 草稿生成
- 去重与状态管理

### 交互层

- 飞书通知
- 后续补充的人工确认 / 发送入口

### 交付层

- 本地 Agent 或专用主机服务
- 配置管理
- 日志与升级机制

这个架构与当前项目一致，也便于以后接入自动发送和客户交付。

## 14. 当前最值得执行的产品动作

1. 从所有产品文档中移除“Web 已可用”的表达，统一口径为“Web 仅保留接口抽象”。
2. 明确当前版本是“Android 草稿辅助版”，不要对外表述为自动回复产品。
3. 把 Android 自动发送列为下一阶段的唯一核心功能。
4. 提前按“外部客户交付”要求补齐安装、排障、设备要求和运行说明。
5. 以“一客户一设备一账号”为默认交付模型设计后续方案。

## 15. 一句话产品方向

先把它做成一个**能在 Android 真机上稳定发现私信并生成 AI 草稿的外部可交付产品**，再在此基础上扩展为**具备受控自动发送能力的小红书私信自动化系统**。
