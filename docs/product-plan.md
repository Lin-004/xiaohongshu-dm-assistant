# 小红书私信 AI 助手产品方案

## 目标

为个人小红书账号建立一条“新私信发现 -> AI 生成回复 -> 可选自动发送 -> 飞书通知本人”的闭环，减少错过消息和手动重复回复的成本。

## 用户价值

- 不错过潜在客户、合作方或粉丝咨询
- 对高频问题实现更快回复
- 即使自动回复，也能通过飞书实时看到发生了什么

## 关键约束

- 首版假设没有稳定的个人账号私信开放 API
- 页面 DOM 可能变化，浏览器自动化需要可调选择器
- 全自动发送存在误判风险，需要默认保守

## MVP 边界

### 做

- 保持网页登录态
- 轮询私信页会话列表
- 识别未读会话
- 提取最近几条消息作为上下文
- 调用大模型生成回复
- 命中安全规则时只通知不发送
- 飞书通知新消息、AI 回复、发送状态

### 不做

- 多账号切换
- 复杂知识库检索
- CRM 管理后台
- 多轮意图编排

## 关键策略

### 回复策略

- 语气像真人，不像客服机器人
- 优先简洁回答
- 不承诺不确定事项
- 涉及价格、退款、投诉、侵权、法律、联系方式时转人工

### 风险控制

- 默认 `AUTO_SEND_REPLY=false`
- 敏感关键词转人工
- 每个会话设置冷却时间
- 本地状态去重

## 技术方案

### 小红书侧

- Playwright 持久化浏览器上下文
- 手动登录后复用会话
- 轮询会话列表与未读标记

### 大模型侧

- 使用兼容 OpenAI Chat Completions 的 HTTP 接口
- 输入最近消息和会话标题
- 输出单条可直接发送的中文回复

### 飞书侧

- 使用自定义机器人 webhook
- 推送摘要、AI 建议、是否自动发送、错误信息

### 状态存储

- `.data/state.json`
- 记录每个会话最后处理的消息摘要 hash 和最后回复时间

## 分阶段落地建议

### Phase 1

监控 + AI 生成 + 飞书通知，不自动发送

### Phase 2

仅对白名单场景自动发送，例如：

- 你好
- 在吗
- 能发下信息吗
- 这个怎么购买

### Phase 3

增加知识库、FAQ、人工接管入口、运营报表
