# Android 消息增量测试指南

## 1. 测试目的

本轮测试用于验证 Android 主链路的两项关键能力是否符合技术设计文档：

1. 候选会话必须按串行方式处理
2. `unread = true` 只是候选信号，真正是否处理要看会话内是否存在“未处理的消息增量”

本轮不测试自动发送，只测试：

- 候选会话发现
- 进入会话
- 上下文读取
- 消息增量识别
- 无增量跳过
- 有增量时生成 AI 草稿并发送飞书通知

## 2. 测试环境

请确认以下条件成立：

- Windows 主机可正常执行 `adb`
- Android 手机已连接并授权 USB 调试
- 小红书已登录目标账号
- 项目 `.env` 已配置完成
- 已确认 `npm start` 可以正常启动监听
- 飞书机器人可以正常接收通知

建议固定使用同一台测试手机、同一账号，避免环境漂移。

## 3. 本轮重点验证点

### 3.1 串行处理

验证系统是否满足以下行为：

- 一次只处理当前列表中的第一个候选会话
- 当前会话处理结束后，返回消息列表
- 返回列表后重新抓取候选队列
- 不复用旧列表快照继续点下一个会话

### 3.2 消息增量识别

验证系统是否满足以下行为：

- 列表里有 `unread` 红点，不代表一定会生成回复
- 进入会话后，如果没有新的有效消息增量，应跳过处理
- 如果有连续多条新消息，应聚合成一次回复输入
- 飞书通知里的“用户消息”应反映聚合后的输入，而不是机械只取最后一条

## 4. 测试前准备

建议先准备 3 类测试会话：

### A 类：有真实新增消息的会话

要求：

- 会话列表有未读标记
- 进入详情页后，确实能看到新消息

用途：

- 验证“有增量时生成草稿”

### B 类：列表有未读，但详情无新增消息的会话

要求：

- 列表中显示未读或红点
- 进入详情后，没有新消息增量，或消息已经属于此前处理过的内容

用途：

- 验证“候选但跳过”

### C 类：同一会话连续多条新消息

要求：

- 同一个用户连续发送 2 到 3 条消息

用途：

- 验证“消息聚合”

如果难以人工制造 B 类场景，可以优先把 A 类和 C 类跑通，再记录当前无法稳定构造的原因。

## 5. 执行方式

### 5.1 调试模式

用于观察单次链路：

```powershell
npm run debug:android
```

重点观察：

- 是否能识别会话列表
- 是否能进入目标会话
- 是否能正确读取上下文

### 5.2 正式监听模式

用于验证完整处理链路：

```powershell
npm start
```

重点观察：

- 日志中的候选会话数量
- 是否出现“候选会话没有新增消息，跳过”
- 是否出现“重新抓取列表后，剩余 X 个未读会话”
- 飞书是否收到通知

## 6. 测试用例

### 用例 1：无候选会话

步骤：

1. 将小红书停留在消息列表页
2. 确保当前没有需要处理的未读候选
3. 运行 `npm start`

预期：

- 日志显示“本轮没有发现未读私信”
- 不发送飞书通知

### 用例 2：单个候选，有新增消息

步骤：

1. 准备 1 个有真实新增消息的会话
2. 运行 `npm start`
3. 观察日志和飞书通知

预期：

- 日志先显示发现 1 个未读会话
- 系统进入该会话
- 生成 AI 草稿
- 飞书收到通知
- 日志显示“仅生成: <会话名>”

### 用例 3：候选会话无新增消息，应跳过

步骤：

1. 准备 1 个列表里显示未读、但详情无新增消息的会话
2. 运行 `npm start`
3. 观察日志

预期：

- 日志显示该会话被识别为候选
- 进入详情页后，不发送飞书通知
- 日志出现：

```text
候选会话没有新增消息，跳过: <会话名>
```

### 用例 4：两个候选，先跳过第一个，再处理第二个

步骤：

1. 准备两个候选会话
2. 第一个候选无新增消息
3. 第二个候选有真实新增消息
4. 运行 `npm start`

预期：

- 系统先进入第一个候选
- 第一个被判定为“无新增消息，跳过”
- 返回列表后重新抓取
- 日志出现：

```text
重新抓取列表后，剩余 1 个未读会话
```

- 然后处理第二个候选并发送飞书通知

### 用例 5：同一会话连续多条新消息聚合

步骤：

1. 让同一用户连续发送 2 到 3 条新消息
2. 运行 `npm start`
3. 查看飞书通知中的“用户消息”

预期：

- 系统只对该会话生成一次草稿
- 飞书通知中的用户消息应包含这批新增消息的聚合内容
- 不应只取最后一条消息作为唯一输入

### 用例 6：处理一个候选后，必须重新抓列表

步骤：

1. 准备至少 2 个候选会话
2. 运行 `npm start`
3. 观察日志顺序

预期：

- 第一个候选处理完成后，不直接按旧顺序点第二个
- 日志必须出现“重新抓取列表后”
- 第二个候选是在重抓后再决定的

## 7. 需要收集的证据

每个测试用例建议收集以下内容：

- 执行时间
- 测试前手机所在页面
- 终端完整日志
- 飞书通知截图
- 如有异常，保留 `.data/android-ui-latest.xml`

如果执行 `debug:android`，也请保留：

- 识别到的候选列表
- 打开会话前后的结果

## 8. 异常记录要求

如果结果不符合预期，请至少记录：

- 用例编号
- 实际结果
- 预期结果
- 是否可稳定复现
- 复现步骤
- 对应日志片段
- 是否附带 `android-ui-latest.xml`
