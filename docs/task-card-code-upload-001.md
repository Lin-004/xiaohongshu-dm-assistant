# 线程任务卡

### 线程
代码上传

### 编号
CU-001

### 目标
基于 `CT-003` 已确认通过的结论，收口当前本地 `main` 工作区中的本轮改动，给出可执行的提交、分支、推送和 PR 方案，并完成本轮上传准备或实际上传动作。

### 范围
- 要做：
  - 核对当前 working tree 中哪些文件属于本轮已验证通过的改动
  - 检查当前分支、工作区、远程跟踪关系和上传前置条件
  - 形成本轮提交边界
  - 给出并执行分支、提交、推送和 PR 处理方案
  - 输出上传前检查结果和最终上传状态
- 不要做：
  - 不跳过 `CT-003` 的测试结论
  - 不新增未经主线程确认的代码修改
  - 不把新的功能开发混入本轮上传
  - 不直接沿用旧的“工作区干净 / 上传已完成”历史结论

### 输入材料
- `docs/thread-status.md`
- `docs/code-testing-ct-003-result.md`
- `docs/code-testing-ct-002-result.md`
- `README.md`
- `package.json`
- `src/config.js`
- `src/index.js`
- `test/channel.test.js`
- `test/monitor.test.js`
- `src/thread-status-project-sync.js`
- `scripts/sync-thread-status.js`
- `test/thread-status-project-sync.test.js`
- `docs/` 下本轮新增的任务卡、结果文档和协作文档

### 输出要求
- 一份上传前检查结果
- 明确列出：
  - 本轮准备纳入上传的文件范围
  - 当前分支与工作区状态
  - 建议的提交拆分方式
  - 实际执行的分支 / 提交 / 推送 / PR 动作
  - 若未完成上传，明确阻塞点

### 明确执行要求

#### 1. 上传前检查

- 确认当前测试前置条件以 `docs/code-testing-ct-003-result.md` 为准
- 重新核对当前仓库状态，不得复用旧的历史上传结论
- 必须明确说明当前是：
  - 在现有分支上继续处理
  - 还是从当前 `main` 切出新的上传分支

#### 2. 提交边界收口

- 需要先判断本轮改动是否应整体上传，还是拆分为多个提交
- 默认优先方案：
  - 在同一上传分支内，按“业务代码改动”和“协作 / 同步脚本改动”分成两个提交
- 如果上传线程判断该拆分不合适，必须说明原因

#### 3. 上传动作

- 若上传条件完整，执行：
  - 分支处理
  - 提交
  - 推送
  - PR 创建或更新
- 若上传线程发现当前工作区中存在不应纳入本轮的文件，不要擅自删除，需在结果中明确标出并回传主线程

#### 4. 结果口径

- 必须明确说明最终属于哪一种：
  - 已完成上传准备，但未实际推送 / 开 PR
  - 已完成推送，但未开 PR
  - 已完成推送与 PR
  - 未完成上传，并给出阻塞点

### 完成标准
- 本轮上传范围清楚
- 当前工作区与分支状态清楚
- 提交拆分方案清楚
- 上传结果或阻塞点清楚
