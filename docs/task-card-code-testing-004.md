# 线程任务卡

### 线程
代码测试

### 编号
CT-004

### 目标
基于 `TP-003` 技术结论和 `CW-003` 实现结果，验收 Android 自动发送 Beta 最小闭环，确认自动发送默认关闭、显式开启、低风险成功发送、失败回退、重复发送保护和高风险强制转人工是否成立，并给出当前 Beta 是否达到本轮目标的结论。

### 范围
- 要做：
  - 执行 `npm.cmd test`
  - 验证自动发送相关本地测试是否通过
  - 做 Android 真机验收，重点验证自动发送成功路径
  - 验证发送失败回退、重复发送保护和高风险转人工
  - 明确哪些结论来自真机，哪些来自代码级 / 受控场景
  - 基于验收结果给出是否达到本轮 Beta 目标的结论
- 不要做：
  - 不重构代码
  - 不扩展到多账号、后台、CRM、纯云端 SaaS
  - 不恢复 Web 通道自动发送
  - 不替代主线程做上传或发布决定

  - 真机通过项
  - 代码级 / 受控场景通过项
  - 失败项
  - 风险项
  - 是否达到当前自动发送 Beta 目标

### 明确测试要求

#### 1. 完整测试套件

- 执行 `npm.cmd test`
- 记录总通过数 / 总失败数
- 重点关注：
  - `test/monitor.test.js`
  - `test/policy.test.js`
  - `test/android-ui.test.js`
  - `test/channel.test.js`
- 如存在失败项，列出失败用例名称和直接原因

#### 2. 自动发送默认关闭

- 在未显式开启 `XHS_AUTO_SEND_REPLY` 时，验证系统不会进入自动发送路径
- 即使 Android 通道具备发送能力，也应保持“草稿 + 通知”模式

#### 3. 低风险自动发送成功路径

- 在显式开启 `XHS_AUTO_SEND_REPLY` 的前提下，做至少一轮真机验收：
  - 当前页面位于会话详情页
  - 草稿为低风险标准回复
  - 输入框和发送按钮可识别
  - 自动发送后能确认消息真实发出
- 测试报告必须明确说明：
  - 是否完成真机成功发送
  - 发送成功的确认依据是什么
  - 输入法 / 设备兼容性是否稳定

#### 4. 失败回退

- 至少覆盖一个失败回退场景，验证：
  - 系统不会把失败误记为成功
  - 会回退到人工通知
  - 会记录失败码
- 如果真机上难以稳定构造失败场景，允许用代码级或受控场景验证，但报告中必须明确标注

#### 5. 重复发送保护与 cooldown

- 验证同一消息增量不会重复自动发送
- 验证命中 cooldown 时不会自动发送
- 验证无新增消息时不会错误推进 cooldown

#### 6. 高风险强制转人工

- 至少覆盖一个高风险场景，验证：
  - 命中敏感 / 报价 / 站外联系方式等规则时
  - 系统强制转人工
  - 不进入自动发送执行器

#### 7. 页面 / 控件保护

- 验证以下场景不会误触发自动发送：
  - 页面不在 `conversation_detail`
  - 输入框缺失
  - 发送按钮缺失
- 如无法全部真机构造，允许组合代码级 / 受控场景验证，但必须写清楚边界

### 完成标准
- 有明确的完整测试结果
- 有明确的真机验收结果
- 有明确的失败项或风险项
- 有明确说明哪些结论来自真机、哪些来自代码级 / 受控场景
- 有明确说明当前是否达到自动发送 Beta 本轮目标
