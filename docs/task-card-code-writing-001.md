### 明确实现要求

#### 1. 结构化消息行

- 在 `src/channels/android-ui.js` 中输出结构化消息行，而不是只输出扁平 `history: string[]`
- 每条消息至少应包含：
  - `text`
  - `direction`：`inbound | outbound | unknown`
  - `kind`：`message | system | timestamp`
  - `sequence`
- 不允许为了“去重”把消息正文压平成唯一文本集合

#### 2. 增量识别

- 在 `src/policy.js` 中重构消息增量识别逻辑
- 默认规则：
  - 从会话尾部向上遍历
  - 只取连续入站消息
  - 遇到出站、系统行、时间行或未知断点立即停止
  - 有 `unreadCount` 时，最多取 `min(unreadCount, 3)` 条
  - 无 `unreadCount` 时，默认只取最后 1 条连续入站消息
- 通知中的“用户消息”必须使用增量聚合结果，不允许继续带出整段历史消息
- 候选无新增消息时：
  - 不生成草稿
  - 不发送业务通知
  - 不错误推进 cooldown 时间戳

#### 3. 标题确认

- 在 `src/channels/android.js` 中把详情页标题作为 canonical title
- 列表标题仅作为点击前定位提示
- 已进入详情页但标题不一致时：
  - 不直接报 fatal error
  - 记录 warning
  - 使用详情页标题继续处理
- 只有在点击后仍停留在列表页时，才判定为真正导航失败

#### 4. 调度最小落地

- `src/index.js` 保持当前串行重抓骨架
- 本轮允许加入两级调度参数入口，但默认最小值固定为：
  - `Tier 1 = 首屏`
  - `Tier 2 = 额外 1 屏`
  - `补偿配额 = 4:1`
- 如果真实深扫接入成本过高，本轮允许先只把参数入口和执行骨架整理好，但不能破坏现有串行重抓流程
