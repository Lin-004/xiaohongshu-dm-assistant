## 1. 结论摘要

本轮推荐**正式采用“两级调度策略”作为 Android 主链路默认模型**，但当前迭代只落地最小稳定版本：

- 调度默认模型：`Tier 1 首屏抢占优先 + Tier 2 单屏补偿`
- 默认补偿配额：`4:1`
- 默认深层补偿扫描范围：`1 屏`
- 默认消息增量窗口：`按会话尾部连续入站消息聚合，优先受 unreadCount 约束，默认上限 3 条`

同时明确：**修复用例 2、5 的前提不是只改参数，而是先把“消息增量识别输入”从扁平 history 字符串改成“有顺序、可区分方向的消息行结构”**。如果仍沿用当前 `history: string[]` 的粗粒度结构，消息增量边界不会稳定，标题识别问题也会继续干扰导航确认。

## 2. 推荐方案

### 2.1 默认调度模型

采用以下默认模型：

- `Tier 1`：当前首屏候选会话，抢占优先
- `Tier 2`：向下额外扫描 1 屏得到的补偿候选
- 每轮处理逻辑：
  1. 先抓首屏候选
  2. 优先处理 Tier 1
  3. 每处理完 4 个 Tier 1 候选，允许处理 1 个 Tier 2 候选
  4. 一旦首屏重新出现新候选，立即中断 Tier 2，回到 Tier 1

### 2.2 为什么采用该默认模型

- 它与当前测试结论一致，优先修复 Android 主链路稳定性，不扩展功能边界。
- 它能兼顾“最新消息响应时效”和“深层旧未读不饿死”。
- 它比全量深扫更适合当前 Android UI 自动化稳定性水平。
- 它能在当前仅能稳定解析首屏约 9 个会话候选的前提下逐步演进。

## 3. 默认参数建议

### 3.1 调度默认参数

建议代码编写线程直接采用下面的默认值：

| 参数 | 默认值 | 说明 |
| --- | --- | --- |
| `tier1VisibleScreens` | `1` | 当前迭代只把首屏作为高优先级区，避免扩大 UI 不稳定范围 |
| `tier2ExtraScreens` | `1` | 每次补偿只额外向下探测 1 屏 |
| `tier1ToTier2Quota` | `4:1` | 每处理 4 个 Tier 1 候选，允许处理 1 个 Tier 2 候选 |
| `tier2MaxCandidatesPerCycle` | `3` | 单轮最多处理 3 个补偿候选，防止主循环被深层旧未读占满 |
| `tier2StopNoNewRounds` | `2` | 连续 2 次滑动无新候选即停止深扫 |
| `tier2MaxScanDurationMs` | `6000` | 单轮补偿扫描最多占用 6 秒 |
| `candidateRescanAfterEachHandle` | `true` | 每处理完一个候选后必须重抓列表 |

### 3.2 消息增量默认参数

| 参数 | 默认值 | 说明 |
| --- | --- | --- |
| `incrementMaxMessages` | `3` | 单次最多聚合 3 条连续用户消息 |
| `incrementDefaultUnreadFallback` | `1` | 无 unreadCount 时默认只取 1 条尾部入站消息 |
| `incrementUseUnreadCountCap` | `true` | 有 unreadCount 时，以 unreadCount 作为上限之一 |
| `incrementRequireInboundTail` | `true` | 只允许从会话尾部连续入站消息构造增量 |
| `incrementIgnoreSystemRows` | `true` | 时间分隔、系统提示、关系状态不参与增量 |

### 3.3 标题确认默认参数

| 参数 | 默认值 | 说明 |
| --- | --- | --- |
| `detailTitleAsCanonical` | `true` | 详情页标题为最终会话标题 |
| `listTitleAsNavigationHint` | `true` | 列表标题只作为点开前的辅助定位 |
| `titleMismatchDowngradeToWarn` | `true` | 已进入详情页但标题不一致时，默认先记 warning，再由上下文确认是否继续 |

## 4. 消息增量默认判定边界

### 4.1 推荐定义

“消息增量”默认定义为：

**会话详情页尾部，连续出现、尚未处理、属于用户入站方向的消息行集合。**

### 4.2 默认判定规则

增量识别必须按以下顺序执行：

1. 从会话尾部向上遍历消息行。
2. 只收集“入站用户消息”。
3. 一旦遇到出站消息、系统行、时间分隔行或无法判定方向的断点，停止继续向上扩展。
4. 若 `unreadCount` 可用，则最多取 `min(unreadCount, 3)` 条连续入站消息。
5. 若 `unreadCount` 不可用，则默认只取最后 1 条连续入站消息。
6. 聚合顺序保持原始时间顺序，不得去重重排。

### 4.3 明确不允许的做法

以下做法本轮禁止继续使用：

- 不允许基于 `context.history: string[]` 的去重后文本数组直接推断消息增量。
- 不允许把“最近 N 条消息文本”直接拼接为增量输入。
- 不允许在无法区分消息方向时，把全部历史文本回填给 LLM。

### 4.4 与用例 2 的直接关联

用例 2 失败的核心原因是：

- 当前“用户消息”抓取了多条历史消息
- 当前实现没有可靠区分“入站增量”和“历史上下文”

因此修复约束必须是：

- **增量输入必须来自结构化消息行，而不是 history 文本列表**
- **通知中的“用户消息”字段必须展示增量聚合结果，而不是全 history**

## 5. 标题识别与导航确认规则

### 5.1 默认规则

- 列表页标题只承担“候选定位”和“点击后二次比对”的作用。
- 详情页标题是会话的唯一 canonical title。
- `state key`、通知展示标题、后续处理日志应优先使用详情页标题。

### 5.2 标题识别失败时的处理规则

#### 场景 A：列表标题可疑，但已成功进入详情页

处理规则：

- 不立即报错中断。
- 使用详情页标题替换列表标题作为 canonical title。
- 记录一次 `title_mismatch_warning`。
- 继续完成会话处理。

#### 场景 B：列表标题可疑，且点击后仍停留在列表页

处理规则：

- 视为真正的导航失败。
- 记录 `click_navigation_failed`。
- 跳过当前候选，继续下一个候选。
- 不终止整轮处理。

#### 场景 C：详情页标题为空或不可解析

处理规则：

- 记录 `detail_title_missing`。
- 当前候选跳过，不生成草稿。
- 返回列表继续后续候选。

### 5.3 与用例 5 的直接关联

用例 5 的根因不是聚合逻辑本身，而是：

- 列表标题把真实会话误解析成摘要文本
- 导致点击后标题校验失败，中断了聚合链路

因此本轮修复约束是：

- **列表标题误识别不能继续作为阻塞性 fatal error**
- **详情页标题应作为最终可信标题**

## 6. 全已读、无新增、标题失败的处理规则

### 6.1 全已读

- 不进入任何会话
- 不调用 LLM
- 不发送通知
- 记录 `no_unread_candidates`
- 直接结束本轮

### 6.2 候选存在，但无新增消息

- 不生成草稿
- 不发送业务通知
- 不更新 `lastHandledMessageHash`
- 不更新 `lastHandledAt`
- 可选更新 `lastSeenConversationFingerprint`
- 返回列表继续下一个候选

说明：

当前实现在“无新增消息”分支下更新 `lastHandledAt` 有较高风险，可能误触发后续冷却策略，建议代码编写线程本轮一并修正。

### 6.3 标题识别失败

- 已进入详情页：降级为 warning，使用详情页标题继续
- 未进入详情页：记错误并跳过候选
- 整轮不可被单一候选标题错误阻断

## 7. 模块边界建议

### 7.1 `src/channels/android-ui.js`

本模块本轮应承担：

- 输出结构化会话候选
- 输出结构化消息行
- 明确区分：
  - 会话标题
  - 会话摘要
  - unreadCount
  - 消息方向（至少分为 `inbound` / `outbound` / `unknown`）
  - 系统行 / 时间行

建议新增或调整输出结构：

```js
{
  title,
  latestMessage,
  messages: [
    {
      text,
      direction: 'inbound' | 'outbound' | 'unknown',
      kind: 'message' | 'system' | 'timestamp',
      bounds,
      sequence
    }
  ]
}
```

### 7.2 `src/channels/android.js`

本模块本轮应承担：

- 列表页恢复与详情页确认
- 标题不一致时的降级处理
- 候选点击失败后的跳过与继续

不建议本模块继续承担复杂增量推断逻辑。

### 7.3 `src/policy.js`

本模块本轮应承担：

- 根据结构化消息行计算消息增量
- 处理 unreadCount cap
- 输出聚合后的增量结果
- 保留冷却和人工审核逻辑

建议把“增量识别”与“冷却判断”拆开，避免两者互相污染。

### 7.4 `src/index.js`

本模块本轮应承担：

- 执行两级调度
- 处理全已读/空增量/标题失败分支
- 每个候选后重抓列表
- 维护处理计数与日志

不建议在 `index.js` 内写入过多 UI 识别细节。

## 8. 错误处理建议

建议代码编写线程统一以下错误类别：

| 错误码 | 含义 | 默认处理 |
| --- | --- | --- |
| `NO_UNREAD_CANDIDATES` | 当前列表无候选 | info 日志，结束本轮 |
| `NO_INCREMENT_MESSAGES` | 候选无新增消息 | info 日志，跳过候选 |
| `TITLE_MISMATCH_WARNING` | 列表标题与详情标题不一致，但已进入详情 | warn 日志，继续处理 |
| `DETAIL_TITLE_MISSING` | 详情页标题为空 | warn 日志，跳过候选 |
| `CLICK_NAVIGATION_FAILED` | 点击后未进入详情页 | error/warn，跳过候选 |
| `LIST_PARSE_FAILED` | 列表无法解析 | 中断本轮并上报 |

原则：

- 候选级错误尽量局部降级，不阻断整轮。
- 页面级错误才允许终止本轮。

## 9. In Scope / Out of Scope

### In Scope

- Android 主链路默认调度模型落地
- 首屏与深层补偿默认参数落地
- 消息增量边界修复
- 列表标题与详情标题确认逻辑修复
- 用例 2、5 对应问题修复

### Out of Scope

- Android 自动发送
- Web 通道恢复
- 多账号
- 后台管理
- 外部交付能力扩展
- 大规模深扫优化

## 10. 给代码编写线程的下一张任务拆解

建议主线程直接给代码编写线程下发以下任务：

### 任务 1：结构化消息行改造

- 在 `src/channels/android-ui.js` 中输出结构化消息行
- 区分 `inbound/outbound/unknown`
- 保留原始顺序
- 不再对消息正文做集合去重

完成标准：

- 能为增量识别提供尾部连续入站消息
- 用例 2 不再把历史消息整体带入“用户消息”

### 任务 2：增量识别改造

- 在 `src/policy.js` 中重写消息增量识别
- 默认规则按“尾部连续入站消息 + unreadCount cap + 最多 3 条”
- 无新增消息时不推进 cooldown 时间戳

完成标准：

- 用例 2 修复
- 用例 3、4 保持通过

### 任务 3：标题确认与导航降级

- 在 `src/channels/android.js` 中把详情页标题设为 canonical title
- 列表标题与详情标题不一致时先降级为 warning
- 仅在未进入详情页时认定为真正导航失败

完成标准：

- 用例 5 不再因列表标题误识别直接中断
- 用例 6 保持通过

### 任务 4：两级调度最小落地

- 在 `src/index.js` 中保留现有串行重抓骨架
- 新增最小两级调度参数入口，但本轮只落地：
  - Tier 1 = 首屏
  - Tier 2 = 额外 1 屏
  - 4:1 配额
- 如果改造量过大，允许先留参数接口，下一小步再接入真实深扫实现

完成标准：

- 不影响当前串行处理逻辑
- 不引入新的主链路回归

## 11. 验收建议

代码测试线程下一轮至少需要验证：

1. 用例 2：单个候选仅 1 条新消息时，通知中的“用户消息”只包含增量，不包含历史消息。
2. 用例 3：候选无新增消息时，正确跳过，且不会影响后续真实新增消息。
3. 用例 5：同一会话多条连续消息能正确聚合，不再被标题识别误差直接打断。
4. 用例 6：处理后重抓列表链路保持稳定。
5. 新增一条检查：无新增消息分支不会错误推进 cooldown。

## 12. 不推荐方案

本轮明确不推荐以下做法：

- 继续基于扁平 history 文本数组推增量
- 继续把列表标题错误视为 fatal navigation error
- 先做全量深扫再回来处理
- 在当前迭代直接扩展自动发送

## 13. 线程结果

### 结论

已完成 TP-001：推荐采用“两级调度 + 结构化消息增量识别 + 详情页标题 canonical 化”的最小稳定方案，并已给出默认参数、错误处理规则、失败用例修复约束和对代码编写线程的任务拆解。

### 关键结果

- 已给出默认调度模型和默认参数
- 已明确消息增量边界和标题确认规则
- 已识别当前实现对用例 2、5 的结构性缺陷
- 已拆出可直接下发给代码编写线程的 4 个实现任务

### 风险/问题

- 若本轮不改结构化消息行，只调参数，则用例 2 大概率仍不稳定
- 若仍把列表标题当成强约束，标题识别误差会继续阻断用例 5

### 建议下一步

- 主线程确认采用本结论作为默认实现方案
- 代码编写线程按“结构化消息行 -> 增量识别 -> 标题确认 -> 最小两级调度”顺序实现

### 需要主线程决策

- 是否允许本轮先把两级调度参数接口落地，而把真实深扫实现拆到下一小步
- 是否直接按本结论给代码编写线程发下一张正式任务卡
