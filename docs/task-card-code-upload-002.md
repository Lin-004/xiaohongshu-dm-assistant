# 线程任务卡

### 线程
代码上传

### 编号
CU-002

### 目标
在用户已要求暂停开发的前提下，基于当前本地 `main` 工作区中的自动发送 Beta 改动、测试改动、协作文档和项目同步脚本改动，完成本轮上传准备，给出可执行的提交、分支、推送和 PR 方案，并明确当前测试口径与剩余风险。

### 范围
- 要做：
  - 核对当前 working tree 中哪些文件属于本轮准备上传的改动
  - 检查当前分支、工作区、远程跟踪关系和上传前置条件
  - 形成本轮提交边界
  - 给出建议的提交拆分方式
  - 给出并在条件满足时执行分支、提交、推送和 PR 处理方案
  - 输出上传前检查结果和最终上传状态
- 不要做：
  - 不跳过当前测试结论边界
  - 不把 `CT-004` 真机未完成说成“已完成”
  - 不新增未经主线程确认的代码修改
  - 不把新的功能开发混入本轮上传

### 输出要求
- 一份上传前检查结果
- 明确列出：
  - 本轮准备纳入上传的文件范围
  - 当前分支与工作区状态
  - 建议的提交拆分方式
  - 当前测试口径
  - 实际执行的分支 / 提交 / 推送 / PR 动作
  - 若未完成上传，明确阻塞点

### 明确执行要求

#### 1. 上传前检查

- 重新核对当前仓库状态，不得复用旧的历史上传结论
- 必须明确说明当前是：
  - 在现有分支上继续处理
  - 还是从当前 `main` 切出新的上传分支
- 必须明确说明当前测试口径：
  - 本地完整测试 `35/35` 通过
  - `CT-004` 真机自动发送验收未完成

#### 2. 提交边界收口

- 需要先判断本轮改动是否应整体上传，还是拆分为多个提交
- 默认优先方案：
  - 在同一上传分支内，按“Android 自动发送 Beta 业务代码改动”和“协作 / 同步脚本 / 状态板改动”拆成两个提交
- 如果上传线程判断该拆分不合适，必须说明原因

#### 3. 上传说明口径

- 提交说明、PR 说明或交付说明中，必须显式写清：
  - 自动发送 Beta 代码已接入
  - 本地测试通过
  - 真机自动发送验收因设备未连接而未完成
- 不允许把当前版本描述为“自动发送真机已验证完成”

#### 4. 上传动作

- 若上传条件完整，执行：
  - 分支处理
  - 提交
  - 推送
  - PR 创建或更新
- 若上传线程发现当前工作区中存在不应纳入本轮的文件，不要擅自删除，需在结果中明确标出并回传主线程

#### 5. 结果口径

- 必须明确说明最终属于哪一种：
  - 已完成上传准备，但未实际推送 / 开 PR
  - 已完成推送，但未开 PR
  - 已完成推送与 PR
  - 未完成上传，并给出阻塞点

### 完成标准
- 本轮上传范围清楚
- 当前工作区与分支状态清楚
- 提交拆分方案清楚
- 测试口径清楚
- 上传结果或阻塞点清楚
