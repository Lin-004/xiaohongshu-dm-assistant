# 线程任务卡

### 线程
技术规划

### 编号
TP-003

### 目标
基于当前产品路线和已稳定的 Android 主链路，收敛“Android 自动发送 Beta”的最小可落地技术方案，明确本轮到底实现到哪一层、哪些风险必须由策略拦截、哪些能力必须延后，避免代码线程直接在高风险能力上失控扩张。

### 范围
- 要做：
  - 明确 Android 自动发送 Beta 的单一推荐方案
  - 明确发送链路的最小实现边界：写入输入框、点击发送、结果确认、失败回退
  - 明确自动发送的触发前置条件和风险策略
  - 明确哪些场景必须强制转人工，不允许自动发送
  - 明确是否需要“默认关闭、显式开启”的配置边界
  - 明确给代码编写线程的文件级改动边界
  - 明确给代码测试线程的验收建议
- 不要做：
  - 不直接改代码
  - 不修改产品优先级
  - 不扩展到多账号、后台、CRM、纯云端 SaaS
  - 不恢复 Web 通道主链路
  - 不把上传 / PR / 发布动作纳入本轮方案

- 一份给代码编写线程可直接执行的任务拆解

  - 自动发送触发前置条件
  - 发送成功确认
  - 发送失败回退
  - 重复发送保护
  - 高风险场景转人工

### 明确规划要求

#### 1. 本轮要回答清楚的核心问题

- 自动发送是否默认关闭，必须通过配置显式开启
- 自动发送是否只覆盖“低风险、单轮、标准回复”场景
- 发送前是否需要额外校验会话状态、输入框状态和草稿内容状态
- 点击发送后如何确认“真的发出去了”，而不是只完成了点击动作
- 发送失败、页面偏移、按钮缺失、发送后无新增状态确认时如何回退
- 发送完成后本地状态、日志和审计最少要记录什么

#### 2. 推荐范围要求

- 推荐方案必须是“最小可落地 Beta”，不能一次性把自动发送做成完整产品
- 必须显式区分：
  - In Scope
  - Out of Scope
- 必须明确哪些能力本轮不做，例如：
  - 多账号自动发送
  - 飞书内人工审批发送
  - 批量回复
  - 复杂会话状态机
  - 远程控制台

#### 3. 风险策略要求

- 必须明确自动发送的拦截条件，至少考虑：
  - 敏感关键词
  - 站外联系方式
  - 报价 / 合作 / 投诉 / 高风险词
  - 草稿为空或明显异常
  - 页面不在可确认的会话详情态
- 必须明确重复发送保护和 cooldown 关系
- 必须明确发送失败后是否继续处理后续会话，还是停止当前轮次

#### 4. 文件级边界要求

- 需要给出代码编写线程的文件级改动边界
- 需要明确哪些模块负责：
  - 自动发送策略判断
  - UI 输入与点击
  - 发送结果确认
  - 状态记录与去重
  - 日志与告警

### 完成标准
- 有单一推荐方案
- 有明确 in scope / out of scope
- 有策略边界和失败回退规则
- 有文件级改动边界
- 有测试验收建议
- 结果可以直接作为代码编写线程下一张任务卡输入

### 当前主线程补充说明
- 用户已明确：本轮先跳过上传代码阶段，直接进入下一个开发任务
