## 1. 单一推荐方案

本轮推荐的**Android 自动发送 Beta 最小可落地方案**是：

**默认关闭、显式开启、仅覆盖低风险单轮标准回复场景的受控自动发送。**

该方案只落地以下四个闭环步骤：

1. 写入输入框
2. 点击发送
3. 发送结果确认
4. 失败回退到人工通知

不把本轮自动发送做成完整产品，更不引入复杂审批、批量发送、多账号或后台控制。

## 2. 结论摘要

### 2.1 自动发送是否默认关闭

结论：**默认关闭，必须通过配置显式开启。**

推荐配置边界：

- `XHS_AUTO_SEND_REPLY=false` 作为默认值
- 只有显式配置为 `true` 才允许进入自动发送路径
- Android 通道即使支持发送，也不得自动绕过该开关

### 2.2 自动发送覆盖范围

结论：**本轮只覆盖低风险、单轮、标准草稿回复场景。**

允许的自动发送场景：

- 单个候选会话
- 当前会话详情页状态明确
- 输入框和发送按钮都可识别
- 草稿非空、长度正常、未命中高风险分类
- 本轮回复属于标准首轮答复，不涉及报价、投诉、站外导流等风险内容

### 2.3 本轮到底实现到哪一层

本轮只实现到以下层次：

1. 发送前会话状态校验
2. 输入框定位与写入草稿
3. 发送按钮点击
4. 发送后结果确认
5. 失败时回退到人工通知
6. 本地状态、日志、审计最小记录

本轮不实现：

- 多轮对话智能发送策略
- 飞书内人工审批发送
- 批量回复
- 复杂会话状态机
- 远程控制台

## 3. In Scope / Out of Scope

### In Scope

- Android 自动发送开关
- 发送前前置校验
- 输入框写入与发送按钮点击
- 发送结果确认
- 发送失败回退
- 重复发送保护
- 高风险场景强制转人工
- 最小状态记录、日志与审计字段

### Out of Scope

- Web 通道自动发送
- 多账号自动发送
- 飞书审批流
- 批量回复
- 后台控制台
- 远程调度
- 复杂自动回复策略编排

## 4. 自动发送触发前置条件

只有同时满足以下条件，才允许进入自动发送：

1. `config.xiaohongshu.autoSendReply = true`
2. 当前通道能力声明 `sendReply = true`
3. 当前页面已确认处于 `conversation_detail`
4. 当前候选存在有效消息增量
5. 当前草稿非空，且通过基础内容校验
6. 当前会话未命中人工审核规则
7. 当前会话未命中重复发送保护和 cooldown 保护

任一条件不满足，都必须退回“仅生成草稿 + 飞书通知”模式。

## 5. 风险策略边界

### 5.1 必须强制转人工的场景

以下场景本轮**不允许自动发送**：

- 命中敏感关键词
- 命中站外联系方式
- 命中报价 / 合作 / 商务 / 投诉 / 退款 / 侵权等高风险词
- 草稿为空
- 草稿明显异常
  - 仅标点
  - 长度过短，低于合理阈值
  - 长度过长，超出常规私信首轮回复范围
- 页面不在可确认的会话详情态
- 输入框缺失
- 发送按钮缺失
- 发送后无法确认发送成功

### 5.2 推荐的草稿基础校验

本轮建议代码实现至少包含以下草稿校验：

- `trim()` 后不能为空
- 长度范围建议 `2 <= len <= 120`
- 不允许只包含表情或标点

## 6. 重复发送保护与 cooldown 关系

结论：**重复发送保护优先于自动发送；cooldown 继续保留，并作为自动发送的硬阻断条件。**

执行顺序建议：

1. 先计算消息增量
2. 再做重复发送判断
3. 再做 cooldown 判断
4. 再做人工审核判断
5. 最后才允许自动发送

规则：

- 若 `lastHandledMessageHash` 与当前待发送消息增量哈希一致，则不发送
- 若命中 cooldown，则不发送
- 无新增消息时不得推进 cooldown

## 7. 发送链路最小实现边界

### 7.1 写入输入框

最小要求：

- 能定位输入框
- 点击输入框
- 清空旧内容
- 写入本轮草稿

若输入框无法定位：

- 记 `SEND_INPUT_NOT_FOUND`
- 退回人工通知
- 不阻断整轮后续候选处理

### 7.2 点击发送

最小要求：

- 能定位发送按钮
- 发送按钮处于可点击状态
- 点击后进入发送确认步骤

若发送按钮无法定位：

- 记 `SEND_BUTTON_NOT_FOUND`
- 退回人工通知

### 7.3 发送结果确认

本轮推荐采用**最小双重确认**：

1. 点击发送后重新抓取会话详情 UI
2. 满足以下任一条件即可判定成功：
   - 草稿文本已以出站消息形式出现在消息尾部
   - 输入框被清空，且消息尾部新增与草稿高度相似的出站文本

如果都不满足：

- 视为 `SEND_CONFIRM_FAILED`
- 不把该会话记为已自动发送
- 转人工通知并记录失败

### 7.4 失败回退

本轮默认失败回退规则：

- 发送失败只影响当前会话
- 当前会话回退为人工通知
- 主循环继续处理后续候选，不终止整轮

例外：

- 若发送失败原因属于页面级异常，例如：
  - 当前页面不再是 `conversation_detail`
  - 连续多次 UI 操作失败
  - 列表 / 详情页状态不可识别

则允许中断当前轮次，并发出异常通知。

## 8. 状态、日志与审计最小记录

本轮最少应记录以下字段：

- `mode`: `draft-only | manual-review | auto-send | auto-send-failed`
- `lastHandledMessageHash`
- `lastHandledAt`
- `lastReplyText`
- `lastSendAttemptAt`
- `lastSendResult`: `success | failed | skipped`
- `lastSendFailureCode`（若失败）

日志至少应包含：

- 会话标题
- 自动发送是否开启
- 是否命中自动发送前置条件
- 发送是否成功
- 失败错误码

## 10. 代码编写线程任务拆解建议

### 任务 1：自动发送开关与参数接入

- 在 `src/config.js` 中补充自动发送 Beta 所需配置
- 默认关闭
- 显式开启才生效

### 任务 2：自动发送策略判断

- 在 `src/policy.js` 中补自动发送前置条件判断
- 把高风险拦截和草稿异常拦截做成硬阻断

### 任务 3：Android 发送执行器

- 在 `src/channels/android.js` 中实现：
  - 输入框写入
  - 发送按钮点击
  - 发送结果确认
  - 错误码输出

### 任务 4：主循环接线与失败回退

- 在 `src/index.js` 中接入自动发送分支
- 成功则记录 `auto-send`
- 失败则记录 `auto-send-failed` 并退回人工通知

## 11. 代码测试线程验收建议

代码测试线程下一轮至少应覆盖以下验收点：

1. 自动发送默认关闭，未显式开启时绝不发送
2. 显式开启后，低风险标准回复场景可成功自动发送
3. 发送成功后，能确认消息已真实发出，而不是只完成按钮点击
4. 发送失败时，能回退到人工通知
5. 重复发送保护生效，不会对同一消息增量重复发送
6. 命中高风险分类时，强制转人工，不自动发送
7. 页面不在详情页、输入框缺失、发送按钮缺失时，不自动发送

## 12. 不推荐方案

本轮明确不推荐：

- 默认开启自动发送
- 不做发送结果确认，只点击按钮就视为成功
- 让高风险消息也进入自动发送
- 发送失败后继续假定成功并写状态
- 一次性把自动发送做成复杂审批 / 后台产品
