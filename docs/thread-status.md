## 1. 使用说明

这份文档用于记录多个线程的当前状态。  
用途：

- 作为主线程的统一收口面板
- 作为子线程回传结果后的汇总位置
- 作为所有线程共享的最新项目状态视图

注意：

- 这不是协作规则文档，协作规则仍以 `docs/thread-coordination-template.md` 为准
- 这不是详细设计文档，不在这里展开长篇分析
- 这里只记录“当前有效状态”

## 2. 当前总目标

在 Android 真机上稳定跑通“小红书私信发现 -> AI 草稿生成 -> 飞书通知 -> 人工发送”主链路，并按未来可交付给外部客户的方式推进产品与实现。

## 3. 当前阶段

上传准备 / Android 自动发送 Beta

- 当前唯一真实可用的主链路仍是 Android 真机，`web` 通道仅保留为后续接口抽象。
- 当前 MVP 明确定义为“Android 监听 + AI 草稿 + 飞书通知 + 人工发送”，不包含 Web 私信处理和 Android 自动发送。
- `TP-002`、`CW-002`、`CT-002`、`CT-003` 已完成；当前主链路和同步链路已达到可继续推进下一轮开发的状态。
- 用户已明确要求：暂停当前开发推进，转入上传准备阶段。
- `TP-003` 已完成，主线程已采纳其单一推荐方案：**Android 自动发送 Beta 采用“默认关闭、显式开启、仅覆盖低风险单轮标准回复场景的受控自动发送”。**
- `CW-003` 已完成，代码线程已实现自动发送 Beta 最小闭环，并回传本地测试结果 `npm.cmd test = 35/35` 通过。
- `CT-004` 当前仅完成“代码级 / 受控场景”验收，真机部分因设备未连接而暂停；因此当前可以进入上传准备，但上传结论必须显式标注“真机自动发送验收未完成”。
- 主线程持续维持一条约束：**本轮只能有一个自动发送业务开关，不接受并行双开关配置继续扩散。**
- 由于上传阶段曾被延后，当前 `main` 上已叠加业务代码、测试、协作文档和项目同步脚本改动，上传线程必须重新收口提交边界，不能复用旧的上传历史结论。

- 继续推进新的测试任务
- Web 私信主链路接入
- 多账号调度、后台管理系统、CRM 和纯云端 SaaS 部署
- 飞书内审批流
