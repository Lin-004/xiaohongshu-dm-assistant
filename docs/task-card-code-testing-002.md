# 线程任务卡

### 线程
代码测试

### 编号
CT-002

### 目标
基于 `TP-002` 技术结论和代码编写线程本轮实现结果，验证“默认通道切换到 Android、无新增消息分支收口、串行重抓链路未回退”是否成立，并给出是否具备进入上传阶段前置条件的结论。

### 范围
- 要做：
  - 验证默认通道切换是否生效
  - 验证用例 3 对应分支是否已稳定收口
  - 验证用例 2、5、6 是否未回退
  - 识别本轮改动引入的回归风险
  - 输出是否具备进入上传阶段前置条件的建议
- 不要做：
  - 不重构代码
  - 不扩展测试到 Android 自动发送
  - 不扩展到 Web 主链路恢复
  - 不替代主线程做最终发布决定

### 明确测试要求

#### 1. 默认通道切换

- 在未设置 `XHS_CHANNEL` 时，验证默认通道为 `android`
- 在显式设置 `XHS_CHANNEL=web` 时，验证仍可覆盖切回 `web`
- 如默认值与文档口径不一致，直接记为失败项

#### 2. 用例 3：候选存在但无新增消息

重点验证以下行为是否同时成立：

- 不调用 LLM
- 不发送业务通知
- 不推进 cooldown
- 返回列表后可继续处理后续真实候选

如果独立场景仍难以稳定人工构造，允许通过组合场景验证该分支，但测试报告必须明确说明：

- 是“独立构造通过”
- 还是“仅在组合场景中验证通过”

#### 3. 回归保护

必须回归以下用例：

- 用例 2：消息增量边界不回退
- 用例 5：标题识别误差不再阻断聚合
- 用例 6：串行重抓链路不回退

#### 4. 参数入口边界

- 验证本轮只引入参数入口，没有实际接入真实 Tier 2 深扫
- 如果发现代码线程超范围实现真实深扫，记为风险项或失败项

### 完成标准
- 有明确通过项
- 有明确失败项或风险项
- 明确说明用例 3 的最终测试结论
- 明确说明是否具备进入上传阶段前置条件
