## 1. 使用目的

这份模板用于让多个线程围绕同一个项目协作，而不是各自独立推进、互相覆盖上下文。

目标：

- 明确每个线程只负责什么
- 明确每个线程从哪里拿输入
- 明确每个线程要输出什么
- 让所有结果最终回收到主线程

## 2. 协作规则

### 2.1 总原则

- 只有一个主线程负责最终决策
- 子线程只处理自己职责范围内的问题
- 子线程不能擅自改变其他线程的结论
- 所有关键结论必须回收到主线程
- 所有线程统一以本文件作为协作约束

### 2.2 主线程职责

主线程只做以下事情：

- 维护项目总目标
- 维护当前优先级
- 给各子线程分配任务
- 回收各子线程产出
- 做最终决策
- 更新本模板中的状态区

主线程不应长时间陷入具体代码细节，否则会失去总控能力。

### 2.3 子线程职责

每个子线程只对自己的输出负责，不直接替代其他线程工作。

例如：

- 代码编写线程负责实现，不负责改产品范围
- 代码测试线程负责验证，不负责重构架构
- 技术规划线程负责方案，不负责直接上线
- 产品规划线程负责边界和优先级，不负责具体代码实现
- 代码上传线程负责整理提交和发布动作，不负责设计决策

## 3. 线程角色分工

### 3.1 主线程

**线程名**：主线程  
**职责**：

- 汇总所有线程的最新结论
- 维护当前项目目标与优先级
- 做需求取舍和执行排序
- 下发下一轮任务

**输入来源**：

- 所有子线程的阶段输出

**输出内容**：

- 最新统一目标
- 当前执行顺序
- 每个线程下一步任务
- 最终确认结论

### 3.2 代码编写线程

**线程名**：代码编写  
**职责**：

- 实现明确的开发任务
- 修改代码
- 补充必要测试
- 说明实现影响范围

**不负责**：

- 重新定义产品目标
- 推翻技术规划结论
- 决定发布策略

**输入来源**：

- 主线程分配的任务
- 技术规划线程给出的实现方向
- 产品规划线程确认的范围

**输出内容**：

- 已完成的代码修改
- 影响文件
- 需要测试验证的点
- 遇到的阻塞问题

### 3.3 代码测试线程

**线程名**：代码测试  
**职责**：

- 验证代码实现是否符合预期
- 运行测试
- 检查回归风险
- 输出缺陷和风险

**不负责**：

- 主动大改代码结构
- 修改需求定义
- 代替主线程做上线判断

**输入来源**：

- 主线程给出的测试目标
- 代码编写线程给出的变更范围

**输出内容**：

- 测试范围
- 测试结果
- 失败项
- 风险项
- 是否建议进入上传/发布阶段

### 3.4 技术规划线程

**线程名**：技术规划  
**职责**：

- 给出模块设计、方案评估和风险分析
- 识别实现边界
- 明确开发顺序和技术约束

**不负责**：

- 直接承担完整代码实现
- 修改产品优先级
- 直接负责上线提交

**输入来源**：

- 主线程给出的技术问题
- 产品规划线程给出的业务目标
- 当前代码现状

**输出内容**：

- 技术结论
- 推荐方案
- 不推荐方案
- 风险清单
- 建议拆解任务

### 3.5 产品规划线程

**线程名**：产品规划  
**职责**：

- 定义目标、范围、优先级、阶段边界
- 明确什么要做，什么不做
- 给出验收标准

**不负责**：

- 具体实现细节
- 替代技术线程做架构设计
- 处理具体代码错误

**输入来源**：

- 主线程的业务目标
- 当前项目阶段
- 技术规划线程反馈的实现约束

**输出内容**：

- 需求边界
- MVP 范围
- 优先级排序
- 验收标准
- 待确认事项

### 3.6 代码上传线程

**线程名**：代码上传  
**职责**：

- 汇总已通过验证的改动
- 检查提交前条件是否满足
- 准备提交说明、发布说明或交付说明

**不负责**：

- 重新定义功能
- 单独决定是否跳过测试
- 改写尚未确认的实现方案

**输入来源**：

- 主线程确认可进入上传阶段
- 代码编写线程给出的变更说明
- 代码测试线程给出的验证结论

**输出内容**：

- 上传前检查结果
- 提交说明草稿
- 发布/交付说明草稿
- 缺失项清单

## 4. 线程输入模板

每次主线程向子线程派任务时，都使用以下模板：

```md
## 线程任务卡

### 目标
[这条线程本轮只解决什么问题]

### 范围
- 要做：
  - [事项 1]
  - [事项 2]
- 不要做：
  - [事项 1]
  - [事项 2]

### 输入材料
- [文件/文档/结论 1]
- [文件/文档/结论 2]

### 输出要求
- [希望线程最终产出的内容]

### 完成标准
- [标准 1]
- [标准 2]
```

## 5. 线程输出模板

每个子线程完成后，都必须按以下格式回传主线程：

```md
## 线程结果

### 结论
[一句话说明本轮完成了什么]

### 关键结果
- [结果 1]
- [结果 2]

### 风险/问题
- [风险或问题 1]
- [风险或问题 2]

### 建议下一步
- [建议 1]
- [建议 2]

### 需要主线程决策
- [待决策事项 1]
- [待决策事项 2]
```

## 6. 主线程汇总模板

主线程在每轮结束后，统一更新：

```md
## 当前总目标
[项目当前唯一最高优先级目标]

## 当前阶段
[例如：MVP / 稳定性修复 / 自动发送 Beta]

## 线程状态
| 线程 | 当前任务 | 状态 | 最近结论 | 下一步 |
| --- | --- | --- | --- | --- |
| 代码编写 | [任务] | 进行中/完成/阻塞 | [一句话] | [下一步] |
| 代码测试 | [任务] | 进行中/完成/阻塞 | [一句话] | [下一步] |
| 技术规划 | [任务] | 进行中/完成/阻塞 | [一句话] | [下一步] |
| 产品规划 | [任务] | 进行中/完成/阻塞 | [一句话] | [下一步] |
| 代码上传 | [任务] | 进行中/完成/阻塞 | [一句话] | [下一步] |

## 最新统一结论
- [结论 1]
- [结论 2]

## 当前阻塞项
- [阻塞项 1]
- [阻塞项 2]

## 下一轮安排
- [安排 1]
- [安排 2]
```

## 7. 推荐协作顺序

建议默认按下面顺序流转：

1. 产品规划线程先定义本轮目标、范围、验收标准
2. 技术规划线程基于产品边界给出实现方案和风险
3. 主线程确认本轮采用方案
4. 代码编写线程按确认方案实现
5. 代码测试线程验证实现结果
6. 主线程确认是否通过
7. 代码上传线程准备提交或交付动作

如果是纯 bug 修复，也可以简化为：

1. 技术规划
2. 代码编写
3. 代码测试
4. 代码上传

## 8. 禁止事项

为了避免线程互相打架，禁止以下行为：

- 两个线程同时定义同一个需求范围
- 两个线程同时修改同一类最终结论
- 代码编写线程绕过产品范围直接扩功能
- 代码上传线程绕过测试结论直接推进
- 子线程把未确认内容当成既定事实发给其他线程



## 9. 当前项目可直接使用的任务分发示例

### 9.1 发给产品规划线程

```md
## 线程任务卡

### 目标
明确 Android 页面判定优化是否进入当前迭代，以及本轮验收标准。

### 范围
- 要做：
  - 给出本轮是否纳入 MVP
  - 定义验收标准
  - 定义不在本轮处理的内容
- 不要做：
  - 不直接写代码
  - 不直接给底层实现方案

### 输入材料
- docs/technical-design.md
- docs/android-page-detection-feasibility.md

### 输出要求
- 一份明确的范围、优先级和验收标准

### 完成标准
- 有 in scope / out of scope
- 有明确验收标准
```

### 9.2 发给技术规划线程

```md
## 线程任务卡

### 目标
给出 Android 页面判定层的推荐设计方案。

### 范围
- 要做：
  - 评估当前页面判定方式
  - 提出显式页面状态方案
  - 给出模块拆分建议
- 不要做：
  - 不直接改代码
  - 不改产品优先级

### 输入材料
- src/channels/android.js
- src/channels/android-adb.js
- src/channels/android-ui.js
- docs/android-page-detection-feasibility.md

### 输出要求
- 一份开发可执行的技术方案

### 完成标准
- 有模块边界
- 有错误处理建议
- 有测试建议
```

### 9.3 发给代码编写线程

```md
## 线程任务卡

### 目标
实现 Android 页面判定层，并补点击后详情页确认。

### 范围
- 要做：
  - 新增页面类型判定
  - 增加点击后的确认逻辑
  - 补必要测试
- 不要做：
  - 不扩展自动发送能力
  - 不改产品范围

### 输入材料
- 产品规划结论
- 技术规划结论
- 当前 Android 通道代码

### 输出要求
- 可运行代码
- 测试说明
- 风险说明

### 完成标准
- 能识别列表页/详情页/未知页
- 点击后有确认逻辑
- 测试可执行
```

### 9.4 发给代码测试线程

```md
## 线程任务卡

### 目标
验证 Android 页面判定改造是否正确且无明显回归。

### 范围
- 要做：
  - 检查功能正确性
  - 检查回归风险
  - 报告失败项
- 不要做：
  - 不重新设计架构
  - 不扩大测试目标

### 输入材料
- 代码改动说明
- 相关测试文件
- XML 样本

### 输出要求
- 一份测试结果报告

### 完成标准
- 有通过项
- 有失败项或风险项
- 有是否建议进入上传阶段的结论
```

### 9.5 发给代码上传线程

```md
## 线程任务卡

### 目标
在实现和测试结论确认后，准备代码提交和交付说明。

### 范围
- 要做：
  - 汇总代码变更说明
  - 检查上传前条件
  - 生成提交说明
- 不要做：
  - 不跳过测试
  - 不新增未确认修改

### 输入材料
- 主线程确认结果
- 代码变更说明
- 测试结论

### 输出要求
- 上传前检查结果
- 提交说明草稿

### 完成标准
- 变更范围清楚
- 测试状态清楚
- 缺失项明确
```

## 10. 建议你怎么用

最低成本的用法是：

1. 把这份模板放进项目里，作为统一规则
2. 主线程每次只给一个子线程发一张任务卡
3. 子线程输出统一按“线程结果”模板返回
4. 主线程把结果更新到“主线程汇总模板”

### 11.拒绝与升级规则
- 如果用户要求超出本线程职责范围，线程可以拒绝执行，并说明应由哪个线程处理。
- 如果用户要求与当前已确认范围、优先级或技术结论冲突，线程可以拒绝直接执行，并要求主线程确认。
- 如果用户要求明显不合理、不可验证、风险过高或会破坏当前协作秩序，线程可以拒绝，并给出原因。
- 拒绝时不要直接中断协作，应返回：
  - 为什么不能做
  - 属于谁的职责
  - 建议主线程如何重新分派

如果你后面想进一步规范，可以再加一份 `docs/thread-status.md`，专门存当前迭代的实时状态。
