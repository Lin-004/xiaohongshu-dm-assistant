## 1. 单一推荐方案

本轮代码实现的**最小可落地方案**为：

**只做“默认通道切换到 Android + 现有串行重抓主链路收口 + 用例 3 保护性修正 + 调度参数入口预留”，不实现真实 Tier 2 深扫。**

这是本轮唯一推荐方案，不建议并行推进更深的调度能力。

## 2. 结论

### 2.1 本轮到底实现到哪一层

本轮只实现到以下层次：

1. 运行时默认通道切换到 `android`
2. 保持现有“处理一个候选 -> 返回列表 -> 重抓列表 -> 处理下一个候选”的串行骨架
3. 把 `TP-001` 中已经确认的消息增量、标题确认、无新增跳过逻辑做稳定性收口
4. 补最小参数入口，但不接入真实深层补偿扫描
5. 为用例 3 给出明确技术处理规则，确保该分支稳定且不污染 cooldown

### 2.2 本轮明确不做什么

本轮不做：

- 不实现真实 Tier 2 深扫
- 不做滚动扫描候选池
- 不做首屏/Tier 2 抢占调度的完整执行器
- 不做 Android 自动发送
- 不恢复 Web 主链路
- 不改产品范围

## 3. 为什么这是最小可落地方案

- 主线程已经确认：当前最高优先级是 Android 稳定性修复，不是能力扩展。
- 用例 2、5 已修复并验证通过，当前剩余未收口项是：
  - 用例 3 最终结论
  - 默认通道仍偏向 `web`
  - 是否具备进入上传阶段的条件
- 在这个节点引入真实 Tier 2 深扫，会新增 UI 滚动、候选去重、候选位置失效、抢占调度等新风险，不符合“先恢复主链路稳定”的原则。

结论：**本轮应停止扩展调度能力，只把当前已验证主链路固化下来。**

## 4. In Scope / Out of Scope

### In Scope

- 默认通道切换到 Android
- 现有串行重抓链路稳定性收口
- 无新增消息分支行为收口
- cooldown 不被无新增分支污染
- 标题确认逻辑保持稳定
- 最小调度参数入口整理
- 用例 3 技术保护与验收条件定义

### Out of Scope

- 真实 Tier 2 深扫实现
- 深层候选去重池
- 抢占式两级调度执行器
- Android 自动发送
- Web 通道恢复
- 多账号 / 后台 / 交付系统

## 5. 默认实现边界

### 5.1 默认通道切换边界

本轮应把默认通道从 `web` 调整为 `android`，但边界如下：

- 只改默认值，不删除 `web` 通道抽象
- 保留显式环境变量覆盖能力
- 文档、配置默认值、运行时行为三者保持一致

推荐默认值：

- `config.channel.provider` 默认值从 `web` 改为 `android`

### 5.2 调度边界

本轮调度实现边界为：

- `Tier 1` 真正执行
- `Tier 2` 只保留参数入口，不接真实扫描

也就是说，本轮代码中允许出现如下内容：

- `tier1VisibleScreens = 1`
- `tier2ExtraScreens = 1`
- `tier1ToTier2Quota = 4`

但本轮不允许出现：

- 实际的列表滑动扫描逻辑
- 深层候选池持久化
- 深扫被抢占后的恢复状态机

## 6. 用例 3 的技术处理建议

### 6.1 技术结论

用例 3 当前不应被定义为“必须稳定人工构造”的阻塞项，而应定义为：

**系统必须具备“候选存在但无新增消息时安全跳过”的稳定逻辑；测试场景若难以独立构造，可接受通过组合场景验证该分支存在且行为正确。**

### 6.2 默认处理规则

当候选会话进入详情页后，若计算结果为“无新增消息”，则：

1. 不调用 LLM
2. 不发送通知
3. 不更新 `lastHandledAt`
4. 不更新 `lastHandledMessageHash`
5. 记录 `NO_INCREMENT_MESSAGES`
6. 返回列表并继续后续候选

### 6.3 为什么这样定义

- 这能避免“无新增消息”分支被 cooldown 污染。
- 这能把用例 3 从“场景构造问题”转化为“分支行为正确性问题”。
- 这符合当前测试现实，不强迫代码线程为人工场景构造问题扩设计。

## 8. 回归风险

### 8.1 主要风险

- 默认通道改为 `android` 后，未配环境变量的运行环境会直接走 Android 链路，若设备未连接会更早失败。
- 若“无新增消息”分支仍更新 `lastHandledAt`，会继续污染 cooldown，导致用例 3 结论不稳定。
- 若代码线程顺手接入真实深扫，容易引入新的导航和列表重排回归。

### 8.2 风险控制建议

- 默认通道切换后，保留环境变量显式覆盖能力。
- 本轮所有改动必须围绕“收口现有链路”展开，禁止顺带扩功能。
- 代码评审时重点检查是否有人把 Tier 2 从“参数入口”偷偷推进成真实实现。

### 9.3 回归风险

建议只有在满足以下条件后，才具备进入上传阶段的前置条件：

1. 用例 1、2、4、5、6 继续通过

4. 无新增消息分支不会污染 cooldown
