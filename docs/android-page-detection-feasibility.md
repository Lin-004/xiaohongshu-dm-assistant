## 1. 背景

当前项目的 Android 主链路依赖 `adb` 连接设备，使用 `uiautomator dump` 抓取前台页面的 UI XML，再由本地解析逻辑提取会话列表和会话上下文。

当前实现已经能够支撑 MVP 目标，即“发现私信、读取上下文、生成 AI 草稿、通知人工发送”。但项目尚未建立显式的页面状态机，也没有形成足够稳定的页面判定与异常恢复机制。为了支持后续开发，需要单独评估“当前页面判定逻辑”是否技术可行、适合什么场景、有哪些失效模式，以及下一步应如何补强。

## 2. 评估范围

本评估聚焦以下内容：

- Android 当前页面的获取方式
- 会话列表页的识别逻辑
- 会话详情页的识别逻辑
- 当前实现的技术可行性与适用边界
- 风险点、失败场景与改进建议

不在本评估范围内：

- LLM 草稿生成质量
- 飞书通知链路
- 自动发送输入框定位与发送确认
- Web 通道能力

## 3. 当前实现概述

### 3.1 页面数据来源

当前实现不依赖截图 OCR，也不读取 Activity 名称或应用内部路由，而是采用 Android 无障碍层级树：

1. 通过 `adb shell uiautomator dump` 导出当前前台页面 XML
2. 通过 `adb exec-out cat` 读取导出的 XML
3. 将 XML 保存到本地，供解析和调试使用
4. 在本地对 XML 节点树进行启发式分析

该方式的优点是：

- 不需要 root
- 与截图识别相比，文本可直接读取，解析成本较低
- 可保留原始 XML 样本，便于回归测试和规则微调

该方式的缺点是：

- 严重依赖 App 当前版本的无障碍树结构
- 对节点位置、可点击属性、文案变化敏感
- 难以保证所有弹窗、异常页、过渡页都能被正确识别

### 3.2 当前页面判定方式

当前代码没有实现统一的页面类型枚举，例如：

- `conversation_list`
- `conversation_detail`
- `home`
- `unknown`

当前逻辑采用的是“隐式页面判定”：

- 当 XML 能够被解析成会话列表结构时，认为当前页是会话列表页
- 当 XML 能够被解析出会话标题和消息正文时，认为当前页是会话详情页
- 当解析失败或结果为空时，通常只能视为页面不匹配、页面异常或规则失效

因此，当前实现本质上不是“先判定页面，再执行动作”，而是“执行解析，如果解析结果像目标页面，就把当前页当成目标页面”。

## 4. 当前页面判定逻辑拆解

### 4.1 会话列表页识别逻辑

会话列表识别基于对 UI 树中“像一行会话卡片”的节点进行筛选。当前主要判定条件包括：

- 节点类型为普通 `node`
- 节点具备 `clickable="true"`
- 节点宽度至少覆盖半屏
- 节点高度大于最小阈值
- 节点位置不能过于靠近顶部或底部
- 节点内部存在“像会话标题”的文本
- 节点内部不能全部都是系统提示、占位文本或无效文案

在识别出候选会话行后，逻辑还会继续：

- 去除嵌套容器导致的重复候选
- 提取标题文本
- 提取摘要文本
- 推断是否存在未读
- 保存候选节点的 bounds，供后续点击进入详情页

这套逻辑本质是基于“几何布局 + 文本特征 + 可点击属性”的启发式规则。

### 4.2 会话详情页识别逻辑

会话详情页没有显式的判断函数。当前做法是直接尝试从页面中抽取：

- 顶部标题
- 消息正文列表
- 最近一条消息

具体思路包括：

- 在页面顶部区域寻找可作为标题的文本
- 在正文区域收集文本节点
- 排除标题、日期、在线状态、未读数、输入框占位、系统提示、底部反应标签等干扰项
- 剩余文本视为消息历史

如果这套规则能抽出合理结果，则流程会继续向下执行；如果抽不出结果，当前实现也没有统一的“详情页判定失败”恢复分支。

### 4.3 页面跳转后的确认逻辑

当前实现点击会话后，只是等待一个短暂时间，再次抓取 UI XML 并读取上下文。这里缺少强校验：

- 没有验证点击后页面是否真的发生变化
- 没有验证当前页是否已进入对应会话详情页
- 没有验证标题是否与点击前的候选会话一致

这意味着以下情况容易被误判：

- 点击无效
- 点击命中了错误区域
- 弹窗覆盖
- 页面切换过慢
- 进入了非预期页面

## 5. 技术可行性结论

## 5.1 结论摘要

当前方案技术上可行，但可行性有明确边界：

- 对于单设备、单账号、人工发送闭环的 MVP，当前实现是可行的
- 对于需要连续稳定运行的内部工具，当前实现勉强可行，但必须补强恢复与校验
- 对于自动发送、无人值守、对外交付场景，当前实现还不够稳，不建议直接作为生产级页面判定方案

### 5.2 为什么说“可行”

当前方案可行，主要基于以下事实：

- Android 无障碍 XML 是真实存在且可持续获取的结构化数据源
- 页面解析与设备交互已做分层，后续可以独立修改规则而不影响 ADB 基础能力
- 当前规则已经覆盖会话列表与会话详情两个核心场景
- 项目支持保存最近一次 XML，用于调试与规则回归
- 项目已有解析相关测试，说明这套思路具备可维护性基础

### 5.3 为什么说“还不够稳”

当前方案不够稳，核心原因是：

- 页面判定是隐式的，没有统一状态机
- 规则高度依赖具体 UI 布局和文案
- 缺少点击后页面确认
- 缺少异常页、弹窗页、空页面、延迟加载页的系统性处理
- 缺少样本库驱动的解析回归体系

## 6. 风险分析

| 风险 | 概率 | 影响 | 说明 |
| --- | --- | --- | --- |
| App UI 改版导致规则失效 | 高 | 高 | 当前规则直接依赖节点结构、位置和文案 |
| 不同设备分辨率或字体缩放导致误判 | 中 | 高 | bounds 阈值与文本布局存在设备差异 |
| 弹窗、引导层、权限提示干扰页面识别 | 中 | 高 | 当前没有统一异常页判定机制 |
| 点击后未成功进入详情页却继续执行 | 中 | 高 | 缺少页面跳转成功校验 |
| `uiautomator dump` 为空或异常 | 中 | 中 | 当前有报错，但恢复路径偏弱 |
| 群聊、特殊会话、系统消息导致误提取 | 中 | 中 | 当前规则更多针对常规私信页面 |

## 7. 适用边界

### 7.1 适合当前方案的场景

- 单设备单账号运行
- 内部使用
- 有人工值守
- 允许解析失败后人工介入
- 以“识别新消息并生成草稿”为主，不直接自动发送

### 7.2 不适合当前方案的场景

- 多设备并发调度
- 无人值守自动处理
- 对识别准确率和稳定性有强 SLA 的业务
- 需要对外规模化交付
- 需要对异常页、弹窗、网络慢、系统浮层具备稳定处理能力的场景

## 8. 推荐改进方案

### 8.1 建立显式页面状态机

建议新增统一页面类型判定层，至少区分：

- `conversation_list`
- `conversation_detail`
- `unknown`
- `blocked_by_popup`

页面判定层应只负责判断“当前是什么页面”，不直接承担业务解析逻辑。后续流程只能在页面状态明确时继续。

建议的调用顺序：

1. 抓取当前 XML
2. 先执行页面判定
3. 根据页面类型进入列表解析或详情解析
4. 如果页面类型未知，则进入恢复或告警流程

### 8.2 增加页面进入确认

在点击会话进入详情页后，建议增加至少一层确认：

- 页面标题已出现
- 页面标题与目标会话标题近似一致
- 正文区域存在消息文本或输入区域

如果确认失败，应执行：

- 再次抓取 XML 重试判断
- 必要时返回上一级
- 超过阈值后输出错误并终止当前会话处理

### 8.3 建立解析样本库

建议把以下 XML 样本纳入回归测试：

- 标准会话列表页
- 标准会话详情页
- 有未读标记的列表页
- 无未读列表页
- 群聊
- 带系统提示的详情页
- 弹窗遮挡页
- 空页面或加载失败页
- 不同分辨率设备样本
- 不同字体缩放样本

样本库目标不是覆盖所有页面，而是覆盖最常见的成功路径和失败路径。

### 8.4 将“识别失败”转为可恢复状态

建议定义统一错误类型，例如：

- `EMPTY_UI_DUMP`
- `UNKNOWN_PAGE`
- `CLICK_NAVIGATION_FAILED`
- `DETAIL_PARSE_FAILED`
- `LIST_PARSE_FAILED`
- `DEVICE_DISCONNECTED`

每类错误应有对应策略：

- 重试
- 返回上一级
- 重启 App
- 记录样本
- 告警并转人工

### 8.5 将几何阈值抽为可配置规则

当前 bounds 阈值和位置阈值建议逐步抽出配置层或规则层，而不是散落在解析代码内部。这样做的收益是：

- 更容易基于样本微调
- 更容易按设备维度做兼容
- 更容易在未来增加版本化规则

## 9. 推荐实现路径

### Phase 1：补页面判定与确认

目标：

- 引入显式页面类型判定
- 补齐点击后的详情页确认
- 建立统一错误类型

交付物：

- 页面判定模块
- 点击后确认逻辑
- 基础失败恢复流程

### Phase 2：补样本与回归测试

目标：

- 建立 XML 样本库
- 扩展解析测试覆盖
- 让每次规则修改都有回归保障

交付物：

- 样本目录与命名规范
- 列表页和详情页回归测试
- 异常页识别测试

### Phase 3：补运行稳定性

目标：

- 补设备异常恢复
- 补 App 不在目标页时的恢复流程
- 输出结构化日志和失败原因

交付物：

- 设备与页面恢复策略
- 结构化日志字段
- 告警阈值建议

## 10. 开发落地建议

建议开发按以下原则改造：

- 先把“页面判定”从“业务解析”中拆出来
- 先补失败检测，再补自动恢复
- 先保证列表页和详情页稳定识别，再考虑自动发送
- 先建立样本回归，再继续堆新规则

不建议直接继续在当前解析函数上追加更多 if/else 来“碰运气适配”页面。这样短期可能有效，但长期会快速失控，最终导致规则互相冲突、难以维护。

## 11. 最终结论

当前 Android 页面判定实现具备明确的技术可行性，足以支撑当前项目的 MVP 和内部验证阶段。它的核心问题不是“方案不可行”，而是“方案尚未产品化”。

换句话说，当前实现已经证明：

- 使用 `adb + uiautomator dump + XML 启发式解析` 识别小红书私信页面是可做的
- 当前代码结构也支持继续演进

但在进入自动发送、无人值守或对外交付之前，必须先补上以下三项基础能力：

1. 显式页面状态判定
2. 点击后的页面进入确认
3. 样本驱动的回归测试与失败恢复

在这三项完成前，建议把当前页面判定逻辑定位为“可运行的 MVP 解析器”，而不是“稳定的生产级页面识别系统”。
